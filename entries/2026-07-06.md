### Scenario

**Question:** A critical `payment-gateway` microservice running on a production Linux server is failing to start after a recent deployment. The only error visible in the CI/CD pipeline logs and initial service logs (accessible via `journalctl -u payment-gateway`) is repeatedly showing: `[ERROR] Failed to initialize logger: Permission denied at /var/log/payment-gateway/application.log`. No application code changes related to logging paths or permissions were part of this deployment, and the previous deployment was successful. 

As a Platform Engineer, you have SSH access to the server with `sudo` privileges. Describe the step-by-step sequence of Linux commands you would use to investigate and diagnose the root cause of this 'Permission denied' error. For each command, explain *why* you are running it and what specific information you are hoping to gather from its output to guide your troubleshooting.

### Architecture

**Question:** You are tasked with designing the logging strategy for a new critical microservice running on a Linux VM. This service will generate two types of logs: operational logs (`operational.log`) and sensitive data logs (`sensitive.log`). Both will be written to a dedicated directory, `/var/log/my-service/`. The service process runs as user `appuser`.

The requirements are:
1.  `operational.log` should be readable by the service user (`appuser`), the system's `adm` group, and a `devops` group for troubleshooting. It should be writable only by `appuser`.
2.  `sensitive.log` must be readable *only* by `appuser` and a `security_auditor` user, and writable *only* by `appuser`. No other users, including `root` directly, should be able to read or modify it without specific elevated processes.
3.  Both log files need to be rotated weekly, compressed, and retained for 30 days. Rotated files of `sensitive.log` must strictly maintain their access controls.
4.  The entire logging mechanism needs to be resilient to high log volume spikes (e.g., 1GB/minute) and maintain the host system's performance.
5.  You need to implement a mechanism to alert if log files exceed certain size limits, if log rotation fails, or if unauthorized access attempts are detected on `sensitive.log`.

As a platform engineer, describe the architectural design for achieving these requirements using standard Linux capabilities (file system structure, permissions, processes, tools). Justify your choices, specifically addressing how you would implement permissions for both current and rotated files, ensure secure archiving, and handle potential performance bottlenecks or failures related to log management in a production environment. Discuss any tradeoffs or alternative approaches you considered for crucial components.

### Coding

**Question:** You're on a Linux production server investigating a suspected issue with a critical background service named `transaction_processor`. This service is expected to run as a non-root user `svc_txn`. You need to quickly assess its state.

Provide the exact Linux commands you would use, in sequence, to accomplish the following, and for each command, briefly explain what you would observe or infer from its output:

1.  **Identify the Process ID(s) (PID) and current resource usage** (CPU%, MEM%) of any running `transaction_processor` processes.
2.  **Determine the full path to the executable file** associated with these processes.
3.  **Examine the file permissions and ownership** of this executable file. Based on the service running as `svc_txn`, explain if these permissions are appropriate (e.g., not writable by `svc_txn` but executable).
4.  **Locate any log files** actively being written to by the `transaction_processor` service. Assume standard log file naming conventions (e.g., ending in `.log`).
