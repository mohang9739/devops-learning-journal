## Interview Practice

### Scenario

**Question:** A critical backend service, `transaction-processor`, is experiencing intermittent failures and generating a high volume of 'java.lang.OutOfMemoryError' messages in its log file, located at `/var/log/transaction-processor/application.log`. You SSH into the affected server.

Describe the sequence of commands you would use to:
1.  Quickly view the most recent 50 lines of the log file and filter them to only show lines containing 'OutOfMemoryError', saving this filtered output to a temporary file named `oom_errors.txt` in your current directory.
2.  After reviewing `oom_errors.txt`, identify the PID of the `transaction-processor` service process.
3.  If you determine the process needs to be restarted, what command would you use to send a graceful termination signal to the `transaction-processor` process?

### Architecture

**Question:** Your team is deploying a new critical service, `backend-processor`, which requires access to a highly sensitive API key stored in `/etc/backend-processor/api_key.conf`. From a security architecture perspective, you need to ensure this file is: 
1. Only readable by the `backend-processor` service process itself. 
2. Writable only by a dedicated `deployer` administrative group. 
3. Completely inaccessible (read/write/execute) to all other users on the system.

Assuming the `backend-processor` service runs as its own dedicated Linux user `backend-processor`, and there's an existing `deployer` Linux group, describe the exact Linux commands you would execute (including file creation/touch, ownership, and permissions) and provide the detailed reasoning behind each step to achieve this secure configuration. How would you verify that your setup correctly implements these security requirements?

### Coding

**Question:** You're investigating an issue with a critical `my_app` service running on a Linux server. The service logs are located under `/var/log/my_app/`, and the main executable is `my_app_service`. A restart script is at `/opt/my_app/restart.sh`.

Please provide the sequence of Linux commands you would use to perform the following diagnostic and corrective actions:

1.  **Locate recent errors:** Find all `.log` files under `/var/log/my_app/` that contain the word "ERROR". Output the paths of these files.
2.  **Extract critical logs:** From the files whose paths were identified in step 1, extract all lines that contain the word "CRITICAL" and save them to a new file named `critical_errors.txt` in your current working directory.
3.  **Identify and manage the service process:**
    a.  Identify the process ID (PID) of the `my_app_service` process.
    b.  If `critical_errors.txt` is not empty (assume you have already checked its content), provide the commands to gracefully terminate `my_app_service`, and then forcefully terminate it if it doesn't respond to the graceful signal.
4.  **Correct script permissions:** Ensure the `/opt/my_app/restart.sh` script is executable by its owner (`appuser`), and read-only for all other users. No one except the owner should have write permissions. Assume `appuser` is already the owner of the script.
